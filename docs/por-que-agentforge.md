# Why AgentForge exists

AgentForge exists because real projects need more than loose prompts and a giant `AGENTS.md`.

The problem is not only “legacy code”. The problem is that context becomes unmanageable:

- `AGENTS.md` and `CLAUDE.md` grow until nobody wants to edit them.
- context, policies, commands, and workflows get mixed together.
- each engine becomes a parallel source of truth.
- humans cannot tell what is canonical and what is derived.
- agents get too much context, or the wrong context.
- new projects start without a clear agent-ready base.
- existing projects accumulate agentic instructions without a shared structure.

AgentForge solves that by analyzing the project, recommending what to add, and organizing the result as a project-local agent-ready layer.

## What it gives you

- `.agentforge/` as the source of truth.
- `analyze` as the project scanner.
- `research-patterns` as the local pattern research step.
- `suggest-agents`, `suggest-skills`, and `apply-suggestions` as the promotion pipeline, with flow and policy suggestions generated by `analyze`.
- `harness/context-index.yaml` as the loading map.
- `skills/` as reusable procedures.
- `flows/` as repeatable workflows.
- `policies/` as the boundaries that keep work safe.
- `compile` as the export step that produces clean bootloaders for Codex, Claude Code, Cursor, Gemini, and GitHub Copilot.

## Who it is for

- teams that want a clean, project-local agent layer
- projects where multiple engines need to share the same source of truth
- humans who need the structure to stay readable over time
- agents that need the right context, not all the context
- projects that need planning, automation, operations, data, knowledge, support, domain, security, compliance, or integration roles, not just engineering roles

## What it is not

AgentForge is not a generic prompt wrapper.
It is not only a static analysis tool.
It is not a spec dump.

It is a system for organizing agent-ready work so the project stays understandable and safe to evolve.
